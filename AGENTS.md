# Codex 작업 규칙

이 파일은 저장소 전체에 적용됩니다.

## 기본 원칙

- 작업 전 `README.md`, 대상 서비스의 `README.md`, 현재 `git status`를 확인한다.
- 기존 파일을 수정하거나 덮어쓰기 전에 내용을 확인하고 사용자 변경을 보존한다.
- 요청 범위 밖의 리팩터링, 삭제, 커밋, 푸시를 수행하지 않는다.
- 운영 경로는 `/srv/stacks/<service>`, 영속 데이터 경로는 `/srv/data/<service>`를 기준으로 한다.
- 서비스 디렉터리는 원칙적으로 `compose.yaml`, `.env.example`, `README.md`를 포함한다.

## 비밀 및 보안

- 실제 비밀번호, 토큰, API 키, 쿠키, 개인 키를 저장하거나 출력하지 않는다.
- 비밀값은 `.env` 또는 외부 비밀 관리 시스템에서 주입하고, Git에는 `.env.example`만 저장한다.
- `.env.example`에는 명백한 자리표시자만 사용하며 실제처럼 보이는 값도 넣지 않는다.
- 새 비밀 파일 형식을 도입하면 같은 변경에서 `.gitignore`를 갱신한다.
- 불필요한 호스트 포트 공개, 특권 모드, Docker 소켓 마운트를 피하고 필요 시 위험을 문서화한다.

## Compose 변경

- 가능하면 명시적인 이미지 버전, 재시작 정책, 헬스체크를 사용한다.
- 영속 데이터는 저장소 내부가 아닌 `/srv/data/<service>` 아래에 둔다.
- 환경 변수는 `${NAME:?message}` 또는 안전한 기본값으로 선언한다.
- 변경 후 `docker compose config`로 구문과 변수 해석을 검증한다. Docker를 사용할 수 없으면 그 사실을 보고한다.
- 서비스별 README에 준비 디렉터리, 필수 변수, 시작 및 복원 방법을 함께 갱신한다.

## 검증 및 인계

- 변경 후 비밀값 포함 여부와 `git diff --check`를 확인한다.
- 마지막에 `git status --short`로 생성·변경 파일을 확인한다.
- 수행한 검증, 수행하지 못한 검증, 운영자가 채워야 할 값을 요약한다.
- 사용자가 명시하지 않는 한 `git commit`, `git push`, 실제 서비스 기동은 하지 않는다.
